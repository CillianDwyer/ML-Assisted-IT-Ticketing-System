Abstract 
IT helpdesks handling large volumes of support requests face a persistent challenge 
in ticket triage. Manual classification of unstructured ticket descriptions is time-
consuming, inconsistent, and prone to error, particularly when request volumes are 
high or when users provide vague or incomplete information. This dissertation 
presents the design, development, and evaluation of a web-based, machine learning-
assisted IT ticketing system that addresses this challenge by automating the 
classification and routing of support requests. 
The system supports three user roles, end users, technicians, and administrators, 
each with a dedicated interface tailored to their operational responsibilities. When a 
support ticket is submitted, the system automatically classifies the description into 
one of 29 distinct issue types using a trained TF-IDF and LinearSVC pipeline, maps 
the predicted category to a specific support team, and assigns an initial priority level. 
An age-based escalation mechanism and SLA state tracking ensure that unresolved 
tickets receive increasing visibility over time. Additional features include threaded 
ticket messaging, private staff communication, file attachments, notifications, and 
role-based analytics dashboards. 
Five machine learning classifiers were evaluated during development, with 
LinearSVC selected for deployment based on its strong performance on high-
dimensional sparse text data, achieving 67.3% accuracy and a macro F1-score of 
0.670 across 29 categories on an unseen synthetic test set. A structured real-world 
classification test demonstrated correct or operationally acceptable outcomes for 
nine of ten submitted tickets, with correct team routing achieved in all nine cases. A 
key finding was that routing accuracy consistently exceeded category-level accuracy 
due to the structure of the team mapping design, providing meaningful operational 
resilience against classification errors at semantically similar category boundaries. 
The system is implemented as a full-stack web application using React, FastAPI, 
SQLite, and scikit-learn, following an iterative development methodology. Evaluation 
confirmed correct behaviour across all major workflows and boundary conditions. 
The project demonstrates that machine learning can be embedded practically and 
effectively into a realistic IT support workflow, contributing directly to routing and 
prioritisation decisions while retaining human oversight at every stage. Limitations 
including the use of synthetic training data and the absence of a formal usability 
evaluation are acknowledged, with real-world data validation and usability testing 
identified as the primary directions for future work.

1. Introduction 
1.1 Project Background 
In modern organisations, even small IT issues can have a big impact on daily 
operations.[1] Issues such as account lockouts, software problems, VPN failures, or 
network outages can prevent employees from completing routine tasks, making 
reliable IT support essential. As organisations become more dependent on digital 
systems, the volume and variety of support requests continue to grow, increasing 
pressure on IT service desks to handle tickets quickly and accurately. 
During my work placement at an IT helpdesk, I observed that many support requests 
were still handled through manual triage. Staff would read ticket descriptions, 
decide what the issue was, and then assign the ticket to the appropriate team. While 
this process works, it can lead to delays, inconsistencies, and incorrect 
categorisation, especially when ticket volumes are high or when users provide 
unclear descriptions. Common requests such as password resets or software 
installations often take time to sort before any real troubleshooting begins. 
This problem is also reflected in both industry and research. Studies show that 
unstructured ticket descriptions and inconsistent language make classification 
difficult,[2][3] while manual processes can reduce efficiency. Machine learning 
approaches, particularly text classification techniques like TF-IDF combined with 
supervised models, have been shown to improve the automatic categorisation of 
support requests.[4][5][6] However, many existing systems still rely heavily on 
rule-based routing and manual input.[7] 
This project is motivated by that gap. It explores how a machine learning-assisted 
ticketing system can support the IT helpdesk process by automatically classifying 
tickets while still providing the key features expected in a real system, such as 
dashboards, ticket assignment, messaging, notifications, and SLA tracking. The aim 
is to combine practical helpdesk functionality with intelligent automation to improve 
efficiency and consistency. 
 
1.2 Project Description 
This project is a web-based, machine learning-assisted IT ticketing system designed 
to support three main user groups; end users, technicians, and administrators. 
End users can submit support tickets, track their progress, and communicate within 
ticket threads. Technicians can view assigned tickets, update statuses, adjust 
categories where necessary, and collaborate through messaging. Administrators have 
full oversight of the system, including ticket triage, reassignment, analytics, and 
overall operational monitoring. 
The system is built using a React frontend [8], a FastAPI backend [9], a SQLite 
database [10], and a trained machine learning model. When a ticket is created, the 
backend processes the title and description, passing the description to the classifier. 
The model classifies the description into one of 29 distinct issue types, covering 
common IT support scenarios such as authentication problems, hardware faults, 
networking issues, and security incidents. Each issue type maps to a specific support 
team and base priority level, allowing the system to route and prioritise tickets

Beyond classification and routing, the system provides a complete ticket

operational metrics. The aim is not only to improve ticket categorisation, but to

1.3 Project Aims and Objectives 
The overall aim of this project is to design, develop, and evaluate an intelligent IT 
ticketing system that uses machine learning to support the classification, routing, 
prioritisation, and management of IT support requests. 
To achieve this aim, the following objectives were defined: 
1. To analyse current IT helpdesk practices and identify opportunities for 
automation within ticket triage and routing. 
2. To design and implement a full-stack web application consisting of a React 
frontend, FastAPI backend, and SQLite database. 
3. To implement secure login and role-based access control for end users, 
technicians, and administrators. 
4. To develop core ticket management functionality including ticket creation, 
assignment, status updates, category updates, and closure. 
5. To train and integrate a machine learning model capable of predicting ticket 
issue types from user-submitted text. 
6. To automate the mapping of predicted issue types to support teams and base 
priority levels.

2. Literature Review 
2.1 Alternative Existing Solutions 
A review of existing IT service management (ITSM) and helpdesk platforms was 
conducted to understand how real-world systems handle ticket submission, 
assignment, tracking, and reporting. This analysis helped identify the core functional 
capabilities expected in modern ticketing systems, while also highlighting limitations 
in how unstructured ticket data is processed. 
Freshservice [11] 
Freshservice provides a modern service desk interface with automation rules, custom 
ticket forms, status tracking, reporting dashboards, and asset management. 
However, its automation depends on manually defined fields and predefined 
categories, meaning it cannot interpret the semantic content of free-text ticket 
descriptions. This leads to misclassification when users provide vague or incomplete 
information, requiring manual correction. 
 
 
Figure 2.1: FreshService UI 
 
ServiceNow [12] 
ServiceNow is a widely adopted enterprise ITSM platform offering workflow 
automation, SLA management, approval chains, and integration with enterprise 
systems. Despite its configurability, it operates primarily through structured 
workflows and predefined business rules. While machine learning features can be 
incorporated, they are not central to its default configuration, limiting its ability to 
automatically classify tickets based on free-text content.

Summary of Commercial Platform Analysis 
Across all three platforms reviewed, a consistent pattern emerges, strong workflow 
management capability combined with a fundamental reliance on structured inputs 
and manual triage for classification. None of the platforms reviewed provides robust 
automated interpretation [7] of unstructured ticket text as a default capability. This 
gap directly motivates the design of this project, which places ML-based 
classification at the centre of the ticket creation workflow rather than treating it as an 
administrative configuration layer. 
 
Academic Work 
Academic research has extensively explored the application of machine learning to IT 
helpdesk systems, particularly for improving the classification of unstructured ticket 
data. A consistent theme across the literature is the effectiveness of classical 
supervised learning models when combined with appropriate text representation 
techniques. 
“A Machine Learning-Based Help Desk System for IT Service 
Management” [2] 
Al-Hawari and Barham (2019) 
Al-Hawari and Barham (2019) demonstrated that classification accuracy improves 
significantly when richer textual input is used, with accuracy increasing from 
approximately 53.8% to 81.4% when ticket descriptions were included alongside 
basic metadata. Their central finding, that input quality has a greater impact on 
performance than model complexity, directly influenced two design decisions in this 
project: it justified basing classification on the full ticket description rather than just 
the title, and it motivated the Uncategorized fallback for low-signal descriptions, 
since filtering poor-quality inputs before classification is more effective than 
attempting to classify every submission regardless of content.  
“Classifying the Unstructured IT Service Desk Tickets Using Ensemble of 
Classifiers” [3] 
Paramesh et al. (2021) 
Paramesh et al. (2021) found that Logistic Regression, Naive Bayes, and Support 
Vector Machines perform effectively with TF-IDF feature extraction, while ensemble 
methods provide only marginal gains at significantly higher computational cost. This 
reinforced the decision to prioritise TF-IDF vectorisation and select from efficient 
linear classifiers, and provided direct supporting evidence for including LinearSVC as 
a candidate model. 
“Automated Assignment of Helpdesk Email Tickets: An AI Lifecycle Case 
Study” [5] 
Agarwal et al. (2020) 
Agarwal et al. (2020) demonstrated that a simple TF-IDF pipeline can produce 
operationally useful results in real-world deployment without requiring deep 
learning or complex feature engineering. This justified the decision to use a TF-IDF 
and LinearSVC pipeline rather than transformer-based approaches, and their

lifecycle framing influenced the decision to design the ML component as an 
integrated part of the ticket workflow rather than a standalone classifier. 
“IT Ticket Classification: The Simpler, the Better” [6] 
Revina et al. (2021) 
Revina et al. (2021) argue that simpler models outperform more complex 
alternatives when combined with effective feature engineering, particularly for high-
dimensional sparse text data. This provided explicit academic support for deploying 
LinearSVC over SGDClassifier despite the latter's marginally higher raw accuracy: 
LinearSVC offers equivalent practical performance with simpler configuration, 
stronger interpretability of its decision boundaries, and a more established track 
record in text classification. Their finding also supported the decision not to pursue 
extensive hyperparameter optimisation, since investment in feature representation 
yields greater returns than investment in model tuning. 
“Question Classification for Helpdesk Support Forum Using Machine 
Learning” [14] 
Harun et al. (2023) 
Harun et al. (2023) found that TF-IDF combined with SVM models outperformed 
alternative approaches including Naive Bayes, corroborating the earlier findings of 
Paramesh et al. for short, user-generated text. While their focus on forum-style 
queries limits direct applicability, the consistency of their findings with earlier work 
strengthens the overall evidence base, and their 2023 publication date confirms that 
classical approaches remain competitive despite the availability of more 
sophisticated NLP models. 
Summary of Findings 
Taken together, these studies establish a consistent evidence base for the technical 
approach adopted in this project. Feature representation, particularly TF-IDF, plays 
a more significant role in classification performance than model complexity. Linear 
SVM-based approaches consistently achieve strong results on helpdesk ticket data. 
Simple, efficient pipelines can achieve operationally useful performance without 
requiring deep learning. However, a notable gap in the existing literature is that 
classification is almost universally treated as an isolated task, none of the studies 
reviewed addresses how a trained classifier should be integrated into a live helpdesk 
workflow, how routing and prioritisation decisions should be derived from 
predictions, or how the system should handle low-confidence classifications safely. 
This gap is the primary motivation for the present project, which treats classification 
not as the end goal but as the first stage of a broader operational workflow. 
 
2.2 TF-IDF (Term Frequency – Inverse Document Frequency) 
A key requirement for this project is converting unstructured ticket descriptions into 
a numerical format that machine learning models can process. This is achieved using 
TF-IDF (Term Frequency–Inverse Document Frequency), a widely used and well-
validated technique [15][16] for text feature extraction that was identified as the 
most appropriate representation method through the literature reviewed above. 

===== PAGE 15 =====

   
 
 8  
 
TF-IDF works by assigning a numerical weight to each word in a document based on 
two factors. Term Frequency measures how often a word appears in a specific 
document, while Inverse Document Frequency measures how rare the word is across 
the entire dataset. Words that appear frequently in a single ticket but infrequently 
across all tickets receive higher weights, allowing the model to focus on informative, 
domain-specific terms rather than generic language that appears everywhere. 
In the context of helpdesk tickets, this behaviour is particularly valuable. Words such 
as "VPN", "printer", "Outlook", or "MFA" will typically receive high weights because 
they appear consistently within specific issue types but not across all tickets. In 
contrast, words such as "issue", "problem", "help", or "not working" appear across 
nearly all categories and receive low weights, preventing them from dominating the 
feature space without contributing discriminative signal. 
The output of TF-IDF is a high-dimensional sparse vector where each dimension 
represents a term in the vocabulary and most values are zero for any given ticket. 
This sparse structure is well-suited to linear classifiers such as LinearSVC [17], 
which are specifically designed to perform efficiently in high-dimensional spaces 
where most feature values are inactive. The alignment between TF-IDF's output 
format and LinearSVC's computational strengths is one of the primary technical 
reasons this combination was selected, a choice that is supported consistently across 
the literature reviewed in Section 2.1. 
The vectoriser used in this project was configured to extract both unigrams and 
bigrams [18], capturing single words and two-word phrases. This was important 
because many IT issue types are characterised by multi-word expressions, "password 
reset", "shared mailbox", "network drive", and "group policy" are more 
discriminative as phrases than as individual words. This configuration decision was 
informed by Agarwal et al.'s observation that richer feature representations improve 
classification performance, and by the domain-specific nature of IT support language 
where compound terms carry significant meaning. 
 
2.3 Machine Learning Model Evaluation and Selection 
A key objective of this project was to identify an appropriate machine learning model 
for classifying helpdesk ticket descriptions. Since ticket data is represented as high-
dimensional sparse vectors after TF-IDF transformation, the selected model needed 
to perform effectively in this setting while remaining computationally efficient for 
integration into a web-based system. The following five models were evaluated, 
representing a range of probabilistic, linear, and ensemble-based approaches. All 
were trained and tested using the same TF-IDF feature representation and evaluated 
using macro F1 score, which provides a balanced measure of performance across 
multiple classes. 
Logistic Regression 
Logistic Regression was included as a strong baseline model [19], widely used with 
TF-IDF features due to its ability to learn linear decision boundaries in high-
dimensional spaces efficiently. It serves as a useful benchmark, more flexible than 
Naive Bayes while remaining computationally efficient. 
Multinomial Naive Bayes 

===== PAGE 16 =====

   
 
 9  
 
Multinomial Naive Bayes is a classic probabilistic model well suited to word-
frequency representations [19]. Its main advantages are simplicity and speed, 
though its assumption of feature independence rarely holds in natural language, 
which can limit performance on more complex ticket descriptions. 
SGD Classifier 
The SGD Classifier is an efficient linear model that scales well to large sparse 
datasets, approximating Logistic Regression or linear SVM through iterative 
optimisation. It supports flexible loss function configuration but can be sensitive to 
parameter tuning, which may affect consistency across datasets. 
Random Forest 
Random Forest was included to provide a non-linear, ensemble-based comparison. 
However, tree-based models are generally less suited to sparse text data as they do 
not naturally handle high-dimensional feature spaces where most values are zero, 
making it a useful reference point rather than a strong candidate. 
LinearSVC 
LinearSVC was included due to the consistently strong performance of SVM-based 
approaches on high-dimensional sparse text data [17]. SVM models maximise the 
separation between classes, which is particularly effective with TF-IDF features, and 
this choice is supported directly by the literature reviewed in Section 2.2. 
 
Model Testing Setup 
All five classifiers were trained on a synthetic dataset of 43,500 labelled ticket 
descriptions and evaluated on a separate unseen set of 7,250 descriptions, testing 
generalisation to different styles of ticket wording rather than memorisation of a 
single dataset. Performance was measured using accuracy, macro F1-score, and 
weighted F1-score, with macro F1 prioritised due to the 29-class nature of the 
problem and the need to treat all issue types equally. The classification task used 29 
issue-type labels, representing common IT support scenarios including 
authentication issues, hardware faults, software problems, networking issues, and 
security incidents. These labels were also used within the system to support team 
routing and priority assignment.

Discussion 
Consistent with existing research, linear models outperformed the non-linear 
alternative[3][6], with Random Forest performing significantly below all four linear 
classifiers. Performance on the unseen dataset was notably lower than on same-
distribution data, confirming that the key challenge is generalisation to different 
phrasing styles rather than pattern memorisation. 
Among the models tested, SGDClassifier achieved the highest score. However, 
LinearSVC was selected as the final model due to its very similar performance 
combined with its simplicity, efficiency, and strong track record in text classification 
tasks, making it well suited to real-time prediction in an API environment. 
Categories with distinctive terminology such as Printer Issue, Wi-Fi Connectivity 
Issue, and Phishing Report achieved stronger results, while categories with 
overlapping vocabulary such as Access Request, Basic Software Issue, and Server 
Down / Service Outage were more difficult to distinguish. 
Overall, the evaluation confirms that the model is most effective as a decision-
support tool rather than a fully autonomous classifier. This aligns with the system 
design, where predictions can be reviewed and adjusted by technicians or 
administrators, improving efficiency and consistency while maintaining human 
oversight.

2.4 Technologies Researched 
A range of technologies were evaluated prior to implementation to select a suitable 
and cohesive technology stack for the development of a full-stack web application, 
ensuring effective integration between the frontend, backend, and machine learning 
components. 
Frontend Technologies 
Three frameworks were considered: React, Angular, and Vue.js. Angular's steep 
learning curve and heavy configuration overhead made it less suitable for a project of 
this scope, while Vue.js has a smaller ecosystem that would have limited available 
tooling for the dashboards and role-based routing required. React was selected [8] 
for its component-based architecture, flexibility, and large ecosystem of well-
maintained libraries for charting, routing, and HTTP communication. Vite was 
chosen over Create React App for its faster build times and lighter configuration. 
Backend Technologies 
FastAPI, Flask, and Express.js were considered. Flask lacks built-in request 
validation, requiring additional libraries to handle schema enforcement. Express.js is 
JavaScript-based, which would have introduced a cross-language service boundary 
given the Python machine learning component. FastAPI was selected [9] as it 
provides automatic request validation through Pydantic and shares the same Python 
runtime as the classifier, allowing the trained model to be loaded directly into the 
backend without any additional service layer. 
Authentication and Security 
Session-based authentication and JWT-based authentication were considered. 
Session-based authentication stores user state on the server, introducing complexity 
that does not align well with a decoupled frontend-backend architecture. JWTs were 
selected [20] as they are stateless, integrate naturally with FastAPI's dependency 
injection system, and support role-based access control across all three user types 
without additional session infrastructure. 
Database Technologies 
SQLite, PostgreSQL, and MongoDB were considered. PostgreSQL would be 
appropriate for a production deployment but introduces unnecessary server and 
configuration overhead for a prototype. MongoDB's document-oriented structure 
does not naturally represent the relational data in this system, users, tickets, 
messages, and notifications have clearly defined foreign key relationships. SQLite 
was selected [10] for its simplicity and zero-configuration deployment, with 
SQLAlchemy [21] providing full relational structure and a straightforward migration 
path to PostgreSQL if needed. 
Machine Learning 
Classical machine learning using scikit-learn and transformer-based deep learning 
using Hugging Face were considered. Transformer models such as BERT 
[22][23][24] introduce significantly greater computational cost, implementation 
complexity, and inference latency, which is impractical for real-time prediction 
within a web API. Existing research confirms that classical pipelines achieve 
operationally useful performance on short text classification tasks without this 

===== PAGE 19 =====

   
 
 12  
 
overhead [5][6]. scikit-learn was selected [25] for its efficient classifier 
implementations, clean TF-IDF integration, and joblib serialisation support. 
 
2.5 Conclusions 
This chapter established the technical and theoretical foundation for the system. 
Commercial ITSM platforms demonstrate strong workflow management capability 
but share a consistent limitation: reliance on structured inputs and manual triage for 
classification of unstructured ticket text. Academic research confirms that TF-IDF 
combined with linear classifiers can address this gap effectively, with feature 
representation playing a greater role in performance than model complexity. 
LinearSVC was selected as the final classifier based on this evidence, and the broader 
technology stack, React, FastAPI, SQLite, and scikit-learn, was chosen to support a 
cohesive, prototype-appropriate implementation. A key gap identified across the 
literature is the absence of guidance on integrating classification into a live helpdesk 
workflow, which this project directly addresses

4.4 Machine Learning Design 
A dedicated machine learning design was required because the system needed to do 
more than simply classify ticket text. The prediction had to operate as part of a live 
support workflow, where the output of the model directly influences ticket 
categorisation, team routing, and initial priority assignment. For that reason, the 
machine learning component was designed as an integrated decision-support 
module within the backend rather than as a standalone experiment. 
The chosen design uses a TF-IDF + LinearSVC pipeline for issue-type classification 
[15][16][17]. The rationale for this approach, including the model comparison and 
technology selection, is covered in Section 2. 
The machine learning design consists of two main stages: training and inference. 
 
Training Stage 
During training, a labelled dataset of ticket descriptions and issue-type categories is 
prepared and cleaned. The cleaning process removes empty rows, ensures that each 
record contains a valid text description and label, and standardises the input format. 


===== PAGE 30 =====

   
 
 23  
 
The text descriptions form the input variable, while the issue-type labels form the 
target variable. The model is trained to predict one of the 29 supported issue types 
used by the ticketing system. 
The issue types are: 
1 Access Request 16 Network Drive Access Issue 
2 Account Compromise 17 Network Outage / Connectivity 
Issue 
3 Account Lockout 18 OS / Boot Issue 
4 Active Directory Issue 19 Password Reset 
5 Backup / Restore Issue 20 Peripheral / Docking Issue 
6 Basic Software Issue 21 Phishing Report 
7 DNS / Network Resolution Issue 22 Printer Issue 
8 Device Performance Issue 23 Security Policy Violation 
9 Disk Space / Storage Issue 24 Server Down / Service Outage 
10 Email Access Issue 25 Software Installation Request 
11 File Server Issue 26 Suspicious Login 
12 Laptop/Desktop Hardware Issue 27 VM / Infrastructure Issue 
13 MFA / 2FA Issue 28 VPN Issue 
14 Mailbox / Email Sync Issue 29 Wi-Fi Connectivity Issue 
15 Malware / Virus Alert   
 
Each label represents a distinct IT support scenario and was chosen because it maps 
naturally to the operational structure of the system. These labels are broad enough to 
cover common organisational IT problems, while being specific enough to support 
routing decisions. For example, security-related issues such as Phishing Report, 
Suspicious Login, and Account Compromise can be separated from infrastructure 
issues such as VM / Infrastructure Issue or Server Down / Service Outage. 
In the training pipeline, ticket text is transformed using TF-IDF vectorisation. The 
vectoriser uses unigrams and bigrams [18], allowing it to capture both single words 
and short phrases. This is important because many IT issue types are indicated by 
multi-word expressions such as “password reset”, “shared mailbox”, “group policy”, 
or “network drive”. The resulting feature vectors are then passed to the classifier, 
which learns the relationship between these weighted text patterns and the issue-
type labels. Once training is complete, the model pipeline is saved so that the exact 
same preprocessing and classification steps can be reused in production. 
Several additional vectoriser settings were chosen to improve the quality of the 
feature representation. Sublinear term-frequency scaling was enabled, which applies 
a logarithmic adjustment to term counts instead of using raw frequencies. This 
prevents a single repeated word, such as a user typing "printer printer printer" out of 
frustration, from dominating the feature vector for that ticket. A minimum document 
frequency of two was also applied, meaning that any term appearing in only one 
ticket across the entire training set was ignored. Terms like this are usually typos, 
user-specific names, or one-off wording, and keeping them would add to the 
vocabulary size without helping the model distinguish between categories. The 
vocabulary was also capped at 40,000 features, which is large enough to include 
every term that appears with meaningful frequency across 43,500 training tickets 

===== PAGE 31 =====

   
 
 24  
 
while keeping the feature space small enough for fast training and inference. English 
stop words such as "the", "and", and "is" were removed so that common function 
words did not take up vocabulary space or add noise to the feature set. Taken 
together, these decisions reflect the finding from the literature review that feature 
representation has a greater impact on classification performance than model 
complexity [6], and that careful vectoriser configuration is more valuable than 
extensive classifier tuning.

Inference Stage 
During normal system operation, the trained model is loaded by the backend and 
applied whenever a user submits a new ticket. The description entered by the user is

categorisation, the backend includes a simple signal check before applying the model. 
If the text is too short, contains too little meaningful content, or lacks enough unique 
terms, the system returns Uncategorized instead of forcing a weak prediction. This is 
an important design choice because it prioritises safe routing over overconfident but 
incorrect classification.

gain practical value from machine learning while still retaining administrative

4.5 Priority and SLA Rule Design 
The system was designed not only to classify and route tickets, but also to represent 
how urgent each issue is and how close it is to breaching service expectations. For 
that reason, two connected rule sets were introduced: priority determination and 


===== PAGE 34 =====

   
 
 27  
 
SLA state calculation. These rules form an important part of the operational design 
because the machine learning category alone is not sufficient for queue management. 
Two tickets may belong to the same technical category, but differ significantly in 
urgency depending on business impact and how long they have remained unresolved. 
The first rule set determines the ticket priority. Priority is initially assigned when the 
ticket is created and is based on the predicted or selected issue type. This gives the 
system an immediate way to identify tickets that are likely to require faster attention. 
For example, high-risk security and infrastructure incidents are treated as more 
urgent than routine requests or lower-impact software issues. This initial priority 
acts as the ticket’s starting urgency level and is visible across the dashboards, 
technician queue, and ticket details view. 
The second rule set determines the SLA state. Rather than storing a fixed urgency 
label only, the system continuously interprets the ticket in relation to time. The SLA 
state is derived from the ticket’s current priority and age, allowing the system to 
indicate whether a ticket is still progressing within an acceptable window, 
approaching risk, or already overdue. This means that urgency in the system is 
dynamic rather than static. A ticket that initially starts as medium or low priority 
may become increasingly important if it remains unresolved for too long. 
This design was chosen because it supports real helpdesk operations more effectively 
than a simple fixed-priority model. In practice, unresolved tickets become more 
serious over time, even when the original issue type was not critical. The rules 
therefore give the system a way to balance two different dimensions of urgency: 
• the inherent seriousness of the issue itself 
• the operational risk created by delay 
Priority Design 
Priority is assigned in two stages. The first stage is base priority assignment, where 
the issue type determines the ticket’s initial level. The second stage is age-based 
escalation, where the ticket’s priority can increase as it remains open. 
The system supports four priority levels: 
• Low 
• Medium 
• High 
• Critical 
Base priority is linked to the issue category. Severe incidents such as Account 
Compromise, Malware / Virus Alert, Network Outage / Connectivity Issue, and 
Server Down / Service Outage begin at Critical because they may affect security, 
multiple users, or core services. Important but less immediately catastrophic issues 
such as Account Lockout, MFA / 2FA Issue, VPN Issue, and OS / Boot Issue begin at 
High. Operational but less severe issues such as Password Reset, Email Access Issue, 
and Wi-Fi Connectivity Issue begin at Medium. Lower-impact requests such as 
Access Request, Printer Issue, and Software Installation Request begin at Low. 

===== PAGE 35 =====

   
 
 28  
 
This rule-based base priority design ensures consistency across the system. Similar 
issue types are treated in the same way, and technicians do not need to assess 
urgency manually for every incoming ticket before it enters the queue. At the same 
time, the design remains transparent because the relationship between category and 
starting priority is fixed and explainable. 
However, a ticket’s urgency should not remain fixed forever. A lower-priority issue 
that is ignored for too long can become operationally serious. To address this, the 
system includes age-based escalation rules. If a ticket remains open beyond certain 
time thresholds, its priority is automatically increased. This means that the queue 
reflects both issue severity and elapsed time. 
The escalation logic is: 
• after 24 hours, priority increases by one level 
• after 48 hours, priority increases by two levels 
• after 72 hours, the ticket becomes Critical 
• closed tickets are treated as Low 
This design ensures that unresolved tickets move upward in importance, reducing 
the risk that lower-priority issues remain permanently buried beneath new arrivals. 
It also improves fairness in the queue because older tickets are progressively 
highlighted rather than being ignored in favour of newer work.

SLA Rule Design 
The SLA rule design complements the priority rules by translating ticket age and 
urgency into a service-state indicator. While priority shows how important a ticket is, 
SLA state shows how close the ticket is to missing its expected handling window. 
Each priority level is associated with a target number of hours: 
• Low = 72 hours 
• Medium = 48 hours 
• High = 24 hours 
• Critical = 12 hours 


===== PAGE 37 =====

   
 
 30  
 
These target windows reflect the expectation that more urgent issues should be 
resolved more quickly. A critical outage or security event should not remain in the 
queue for the same amount of time as a routine request or minor software issue. 
Using these targets, the system calculates the current SLA state of every ticket. If a 
ticket is closed, its SLA state is treated as met, because no further active handling is 
required. If a ticket is still open, its age is compared with the SLA target associated 
with its current priority. 
The system uses three active SLA conditions: 
• on_track 
• at_risk 
• breached 
A ticket is considered: 
• on_track when it is still comfortably within the target window 
• at_risk when it has reached at least 75% of the target time 
• breached when it has reached or exceeded 100% of the target time 
This design allows urgency to be communicated visually and operationally before the 
ticket actually breaches. The at risk state is particularly useful because it gives 
technicians and administrators advance warning that a ticket needs attention soon, 
rather than only highlighting problems after they have already become overdue.

Interaction Between Priority and SLA 
The two rule sets are intentionally connected. Priority affects the SLA target, and 
ticket age can affect priority. This creates a dynamic relationship between severity, 
elapsed time, and queue visibility. 
For example, a ticket may begin as Medium priority with a 48 hour target. If it 
remains unresolved long enough, its priority may escalate to High, which in turn 
shortens the effective expectation for how urgently it should be handled. As a result, 
the ticket becomes more visible in technician and admin dashboards through both its 
priority badge and SLA state. 
This interaction supports proactive queue management. Instead of relying only on 
static labels, the system constantly re-evaluates the operational condition of each 
ticket. Tickets that are severe from the start are highlighted immediately, while 
tickets that become problematic over time are also surfaced before they are forgotten. 
Operational Value of the Design 
The priority and SLA rule design improves the usefulness of the system in several 
ways. First, it helps technicians focus on the work that is most urgent rather than 
simply processing tickets in arrival order. Second, it gives administrators visibility 
into queue health by showing which tickets are critical, unassigned, at risk, or 
breached. Third, it makes the behaviour of the system more transparent, because 
priority and SLA are determined by defined business rules rather than just human 
judgement. 
Most importantly, these rules ensure that the ticketing system does not only record 
support requests, but actively supports decision-making. The machine learning 
model identifies the likely issue type, the routing logic sends the ticket to the 
appropriate team, and the priority/SLA design ensures that both severity and elapsed 
time are reflected in how the ticket is managed. Together, these mechanisms make 
the system more useful as an operational support platform rather than a passive 
ticket log. 
 
4.6 Messaging, Notifications, and Attachment Design 
The system was designed to support communication and workflow awareness 
throughout the lifecycle of a ticket. In a practical IT support environment, ticket 
handling involves more than simply storing issue details and updating statuses. 
Users and staff need to exchange updates, staff may need to collaborate internally, 
and participants need to be informed when important events occur. For this reason, 
the design included dedicated messaging, notification, and attachment features as 
part of the wider ticket workflow. 
The messaging design allows each ticket to function as a communication thread. 
Public ticket messages enable end users and support staff to exchange updates 
directly within the context of a ticket, preserving the history of the conversation 
alongside the ticket record itself. This approach was chosen because it keeps 
communication attached to the relevant issue, rather than separating it into external 
channels such as email or informal messaging.

 33  
 
In addition to public communication, the design also includes private assist 
messages. These are intended for internal communication between technicians and

4.7 User Interface Design 
The user interface was designed around the principle that different users interact 
with the system for different purposes and therefore require different views, controls, 
and information emphasis [29]. Rather than presenting a single generic ticket-
management screen to every user, the interface design is role-based. This means that 
the structure of the application, the information shown, and the actions available are 
adapted according to whether the user is an end user, technician, or administrator. 
This design decision was made to improve usability and reduce unnecessary 
complexity. In a ticketing system, the needs of a person submitting a support request 
are very different from the needs of a technician working through an assigned queue, 
or an administrator managing team performance and ticket distribution. If all users 
were given the same interface, the result would be cluttered, confusing, and 
operationally inefficient. By separating the interface by role, the system can show 
each user the most relevant information while hiding functions that are unnecessary 
or inappropriate for that role. 
End User Interface Design 
The end-user interface was designed to prioritise simplicity, clarity, and reassurance. 
End users are not expected to understand technical routing rules, issue categories, or 
SLA logic in detail. Their primary goals are to submit a support request, monitor 
progress, and communicate with support staff when necessary. For this reason, the 
user-facing design focuses on straightforward interaction rather than operational 
detail. 
The ticket submission page provides a simple form with fields for title and issue 
description. The design encourages users to describe the problem clearly by 
providing prompts and examples, but it does not require them to understand how the 
machine learning classifier works internally. Once submitted, the system handles 
categorisation, routing, and starting priority automatically. This keeps the form 
simple while still feeding the backend enough detail to support intelligent processing. 
The user also has access to pages for viewing previously submitted tickets and 
opening an individual ticket conversation. In these views, the design emphasises 
visibility of status, assigned team, category, priority, and message history. The ticket 
details page presents the conversation as a clear threaded workspace so that users 
can follow replies and contribute updates without navigating through multiple

disconnected screens. This supports transparency and reduces uncertainty by 
allowing users to see that the issue is progressing through the support process. 
Technician Interface Design 
The technician interface was designed around queue management and ticket 
resolution. Technicians need rapid access to assigned tickets, clear awareness of 
urgency, and efficient controls for updating ticket state. As a result, the technician 
dashboard is more operationally focused than the end-user view. 
The technician queue page presents tickets in a structured table with filters, priority 
indicators, SLA indicators, and quick controls for status and issue-type updates. This 
allows technicians to review and process work efficiently without opening every 
ticket individually. Priority and SLA badges are particularly important in this view 
because they support triage by highlighting the tickets that are closest to breaching or 
are already overdue. 
Technicians also require access to the ticket details workspace, where they can read 
the original request, view the full conversation history, send replies, attach files, and, 
where necessary, send private assist messages to other staff members. The design of 
this workspace supports both customer-facing communication and internal 
collaboration. This is important because technicians may need to ask another 
specialist for input without exposing that internal conversation to the end user. 
The technician interface therefore balances breadth and depth: the queue view 
supports efficient overview and prioritisation, while the ticket details page supports 
deeper investigation and communication. 
Administrator Interface Design 
The administrator interface was designed to support system-wide oversight rather 
than individual ticket resolution. Administrators need visibility of the entire support 
operation, including assignment gaps, queue pressure, SLA breaches, technician 
workload, and ticket distribution across teams and categories. For this reason, the 
administrator design includes both control-oriented and analytics-oriented elements. 
The admin dashboard provides a central operational view of all tickets. It supports 
searching, filtering, assignment review, category adjustment, status updates, and 
visibility of whether tickets are assigned or unassigned. This enables administrators 
to manage triage and intervene when automatic routing or technician ownership 
needs adjustment. 
In addition to ticket tables, the admin interface includes analytics components such 
as charts, metric cards, and summary counts. These visual elements show status 
distribution, issue-type distribution, team workload, critical ticket counts, SLA 
breaches, and other operational indicators. The purpose of this design is not just to 
display tickets, but to give administrators insight into the health of the support 
process as a whole. This helps the interface serve both a supervisory and a decision-
support function.

4.8 Authentication and Security Design 
The system required a secure yet lightweight authentication mechanism appropriate 
for a role-based web application. As part of the overall frontend-backend architecture 
described earlier, JWT-based authentication was selected [20]. When a user logs in, 
the backend verifies the submitted credentials against securely stored password 
hashes, generates a JSON Web Token (JWT), and returns it to the frontend. The 
frontend stores this token and includes it in the Authorization header of subsequent 
requests to protected API endpoints. This enables the backend to identify and 
authenticate users without maintaining server-side session state. 
This approach was chosen because it supports stateless authentication, integrates 
naturally with FastAPI, and aligns well with a decoupled frontend-backend 
architecture in which communication takes place through HTTP requests. It also 
reduces server-side complexity, which is appropriate for a prototype system. In 
practice, this design allows the React frontend to act as a client application while the 
FastAPI backend centrally enforces authentication and authorisation rules for all 
protected workflows. 
Role-based access control (RBAC) forms a core part of the security design. Although 
all users authenticate through the same mechanism, their permissions differ based 
on role. End users are restricted to creating and viewing their own tickets, 
technicians can manage assigned tickets and collaborate internally, and 
administrators have broader access to queue management, analytics, and system-
wide operations. This separation ensures that users can only access functionality 
relevant to their responsibilities. 
Access control is enforced at two levels. At the frontend level, protected routes 
prevent unauthenticated users from accessing internal pages and guide users toward 
role-appropriate views. At the backend level, protected API endpoints enforce 
security more reliably by validating the JWT and checking the user’s role before 
processing any request. This ensures that even if frontend restrictions are bypassed, 
unauthorised actions cannot be executed. The backend therefore acts as the final 
authority on authentication and authorisation. 
Additional considerations include token handling and request security. Tokens are 
used only for authenticated requests and are issued with an expiry time of 60 
minutes, reducing the risk of long-lived token misuse. By relying on token-based 
authentication rather than server-side sessions, the system avoids maintaining per-
user session state while still ensuring secure request validation.

4.9 Data Design 
The system required a relational data structure capable of supporting user 
authentication, ticket ownership, technician assignment, message history, 
notifications, and lifecycle tracking. Because the platform is centred on the 
movement of tickets through a support process, the database design needed to 
preserve operational relationships, ticket lifecycle timestamps, and communication 
history over time. For this reason, the data model was designed around four core 
entities: User, Ticket, TicketMessage, and Notification [30]. 
The Ticket entity is the central object in the system. Each ticket represents a support 
request raised by a user and contains the key business data needed to manage that 
request, including title, description, issue category, assigned team, status, priority, 
timestamps, and ownership fields. A ticket is linked to the user who created it and 
also linked to a technician responsible for resolving it. This dual relationship is 


===== PAGE 46 =====

   
 
 39  
 
important because the system must distinguish between the person requesting 
support and the staff member assigned to provide it. The ticket entity therefore acts 
as the main point through which workflow, routing, and reporting logic are 
organised. 
The User entity stores authentication and role information for all system users. This 
includes end users, technicians, and administrators. In addition to login credentials, 
each user record contains a role attribute, allowing the system to enforce role-based 
access control at both backend and frontend level. Technician users also contain a 
speciality value, which is used to align staff with support teams such as Service Desk, 
Desktop Support, Network Team, Systems Team, or Security Team. Through its 
relationships to tickets and notifications, the user entity supports both participation 
in the support workflow and visibility of relevant system events. 
The TicketMessage entity supports communication within the lifecycle of a ticket. 
Each message belongs to a specific ticket and records the sender, message content, 
timestamp, and optional recipient information for private assistance messages. The 
design also supports attachments by storing attachment metadata such as file name, 
type, path, and size. This allows the ticket details view to function as a 
communication workspace rather than a simple comment log. By modelling 
messages as a separate entity, the system preserves the full history of interaction 
around each ticket, including end-user updates, technician responses, internal staff 
collaboration, and shared files. 
The Notification entity stores system-generated updates related to ticket activity. 
Notifications are linked to a user and, where relevant, to a specific ticket. This allows 
the system to alert users when meaningful events occur, such as new messages, ticket 
assignment, or status changes. Each notification records its type, content, read 
status, and creation timestamp. This design supports asynchronous awareness across 
the application and ensures that important ticket events can be surfaced in overview 
pages and dashboards even when the user is not actively viewing the relevant ticket. 
Together, these four entities support the complete ticket lifecycle from creation to 
closure. A user creates a ticket, the ticket is categorised and routed, a technician may 
be assigned, messages and attachments are exchanged during resolution, and 
notifications are generated as events occur. Because each step is represented through 
explicit relationships in the database, the design maintains traceability of ownership, 
assignment, communication, and status change over time. This relational structure 
therefore provides the foundation required for both the operational workflow of the 
system and the reporting and analytics features built on top of it.

4.10 Design Decisions and Trade-Offs 
A number of design trade-offs were made during the project: 
• SQLite instead of a larger DBMS [10] 
SQLite was chosen because it is lightweight and suitable for a prototype, even 
though a larger production system would likely use PostgreSQL or MySQL. 
• Classical ML instead of deep learning 
A TF-IDF + LinearSVC pipeline [6] was selected because it offers strong 
performance on sparse text data and is realistic to train, test, explain, and 
integrate within the project scope. 
• Integrated ML within backend rather than separate ML 
microservice 
The ML model was loaded directly by the backend to keep the architecture 
simple and reduce deployment complexity. 
• Role-based dashboards rather than a single generic dashboard 
This increased implementation work, but produced a clearer and more usable 
system for the different stakeholder groups. 
These decisions reflect the project’s goal of building a practical and coherent working 
prototype rather than a large-scale enterprise platform.

5.2 Software Development 
The system was implemented as a full-stack web application using React [8] for the 
frontend, FastAPI [9] for the backend, SQLite [10] for persistence, and 
Python/scikit-learn [25] for the machine learning component. Development 
followed an incremental approach. The initial implementation focused on user 
authentication and ticket CRUD operations, after which more advanced features 
were added in stages, including role-aware routing, machine learning classification, 
messaging, notifications, attachments, SLA logic, and analytics dashboards. 
The backend was implemented first because it defined the core business rules and 
data structures used by the rest of the system. SQLAlchemy models were created for 
users, tickets, ticket messages, and notifications [21]. These models established the 
relationships required for ticket ownership, technician assignment, threaded 
communication, and event tracking. Pydantic schemas were then added to validate 
incoming request data and standardise API responses. 
Authentication was implemented using password hashing and JWT-based access 
tokens [20]. This provided a secure mechanism for login while also supporting role-
aware access control in later endpoints. After authentication, ticket workflows were 
implemented through REST endpoints for ticket creation, retrieval, assignment, 
status updates, and category updates. The ticket entity was gradually expanded 
during development to include timestamps, assigned team, priority, SLA state, and 
technician ownership, allowing the full lifecycle of a support ticket to be tracked. 
The frontend was developed in parallel once the core API structure was stable. React 
components were organised into reusable pages and interface modules, with 
protected routes used to restrict access to authenticated users and role-specific 
routes used to separate administrator and technician functionality. Axios was used as 
the communication layer between frontend and backend, with a request interceptor 
automatically attaching JWT tokens to authenticated requests. 
The machine learning component was developed as a separate experimental 
workflow before being integrated into the application. Candidate models were 
compared using external ticket datasets, after which a TF-IDF and LinearSVC 
pipeline was selected based on its strong classification accuracy and fast inference 
time on short text inputs. The trained model was stored using joblib and loaded 
inside the backend at runtime. Ticket descriptions submitted by users are passed to 
this model, which predicts an issue type used to derive the responsible support team 
and a base priority level.

5.5 Machine Learning Development 
The machine learning component was implemented in two stages: model 
experimentation and backend integration. During experimentation, multiple text 
classification models were compared using ticket datasets. The training pipeline used 
TF-IDF vectorisation on ticket descriptions and evaluated several classifiers, 
including LinearSVC, Logistic Regression, Naive Bayes, SGDClassifier, and Random 
Forest. LinearSVC was selected for deployment due to its consistently high accuracy 
on short text classification tasks [6][17], its computational efficiency at inference 
time, and its suitability for linearly separable category boundaries which are typical 
in IT support ticket data. 
The trained model was saved to disk using joblib [25] and loaded once at application 
startup. During integration, the backend inference module was designed to be fault-
tolerant. Rather than forcing a prediction on every input, the system first validates 
whether the ticket description contains enough meaningful content to classify 
reliably. Listing 5.3 shows the full inference module including model loading, text 
normalisation, signal validation, and prediction. 
MIN_DESCRIPTION_WORDS = 3 
MIN_UNIQUE_TOKENS     = 2 
MIN_ALNUM_CHARS       = 8 
  
classifier = None 
try: 
    classifier = joblib.load( 
        os.path.join(BASE_DIR, "ticket_classifier.pkl") 
    ) 
except Exception as e: 
    print("ML model failed to load:", e) 
  
def _normalize_text(text: str) -> str: 
    return re.sub(r"\s+", " ", (text or "").strip()) 
  
def _has_enough_signal(text: str) -> bool: 
    tokens = re.findall(r"[A-Za-z0-9]+", text.lower()) 
    if len(tokens) < MIN_DESCRIPTION_WORDS: return False 
    if len(set(tokens)) < MIN_UNIQUE_TOKENS: return False 
    return sum(ch.isalnum() for ch in text) >= MIN_ALNUM_CHARS 
  
def predict_category(text: str): 
    if classifier is None: 
        return "Uncategorized" 
    normalized = _normalize_text(text) 
    if not _has_enough_signal(normalized): 
        return "Uncategorized" 
    return classifier.predict([normalized])[0] 
Listing 5.3: ML inference module showing fault-tolerant model loading, text normalisation, signal 
validation, and category prediction with Uncategorized fallback.  
This implementation choice was important because real ticket descriptions are often 
incomplete or inconsistent. The three-part validation check, minimum word count, 
minimum unique tokens, and minimum alphanumeric character count, ensures that 
only descriptions with sufficient content are passed to the classifier. If any check 
fails, or if the model is unavailable, the ticket is marked as Uncategorized rather than 
receiving a low-confidence prediction that could misroute it.

allowing older unresolved tickets to escalate over time. As shown in Listing 5.5, the 
compute_ticket_priority function first determines a base priority from the issue 
type, then applies age-based escalation thresholds. A ticket’s priority increases as it 
remains unresolved: after 24 hours it increases by one level, after 48 hours by two 
levels, and after 72 hours it becomes Critical. In addition, tickets already classified as 
High escalate to Critical after 12 hours. 
PRIORITY_THRESHOLDS_HOURS = { "low": 72, "medium": 48, "high": 24, "critical": 12 } 
PRIORITY_ORDER = ["Low", "Medium", "High", "Critical"] 
  
ISSUE_TYPE_BASE_PRIORITY = { 
    "Password Reset":                    "Medium", 
    "Email Access Issue":                "Medium", 
    "Laptop/Desktop Hardware Issue":     "Medium", 
    "VPN Issue":                         "High", 
    "OS / Boot Issue":                   "High", 
    "Server Down / Service Outage":      "Critical", 
    "Network Outage / Connectivity Issue": "Critical", 
    "Malware / Virus Alert":             "Critical", 
    "Account Compromise":                "Critical", 
} 
  
def compute_ticket_priority(*, status, category, created_at) -> str: 
    if not status or status == "Closed": 
        return "Low" 
    age_hours = 0.0 
    if isinstance(created_at, datetime): 
        age_hours = (datetime.utcnow() - created_at).total_seconds() / 3600.0 
    priority = ISSUE_TYPE_BASE_PRIORITY.get((category or "").strip()) 
    if priority is None: 
        team = resolve_ticket_team(category) 
        priority = TEAM_BASE_PRIORITY.get(team, "Low") 
    # Age-based escalation 
    if age_hours >= PRIORITY_THRESHOLDS_HOURS["low"]: 
        return "Critical" 
    base_idx = PRIORITY_ORDER.index(priority) 
    if age_hours >= PRIORITY_THRESHOLDS_HOURS["medium"]: 
        return PRIORITY_ORDER[min(base_idx + 2, len(PRIORITY_ORDER) - 1)] 
    if age_hours >= PRIORITY_THRESHOLDS_HOURS["high"]: 
        return PRIORITY_ORDER[min(base_idx + 1, len(PRIORITY_ORDER) - 1)] 
    if age_hours >= PRIORITY_THRESHOLDS_HOURS["critical"] and priority == "High": 
        return "Critical" 
    return priority

SLA state is derived from the same priority and age logic. Once the effective priority 
is known, the system calculates what proportion of the allowed resolution window 
has elapsed and classifies the ticket as on-track, at-risk, or breached accordingly, 
as shown in Listing 5.6. 
SLA_TARGET_HOURS = { "Low": 72, "Medium": 48, "High": 24, "Critical": 12 } 
  
def compute_ticket_sla_state(ticket: models.Ticket) -> str: 
    if ticket.status == "Closed": 
        return "met" 
    if not isinstance(ticket.created_at, datetime): 
        return "on_track" 
    effective_priority = ticket.priority or compute_ticket_priority( 
        status=ticket.status, 
        category=ticket.category, 
        created_at=ticket.created_at 
    ) 
    target_hours = SLA_TARGET_HOURS.get(effective_priority, 48) 
    age_hours = ( 
        datetime.utcnow() - ticket.created_at 
    ).total_seconds() / 3600.0 
    ratio = age_hours / target_hours if target_hours > 0 else 0 
    if ratio >= 1.0:  return "breached" 
    if ratio >= 0.75: return "at_risk" 
    return "on_track" 
  
def set_ticket_sla_state(ticket: models.Ticket): 
    ticket.sla_state = compute_ticket_sla_state(ticket)

5.11 Challenges During Development 
Several significant challenges arose during implementation as the project developed 
from a simple ticketing prototype into a more complete workflow-driven system.  
A major challenge was dealing with low-quality ticket descriptions and the risk of 
unreliable machine learning predictions. In practice, users do not always submit 
well-formed support requests. Some descriptions were too short, too vague, or too 
lacking in meaningful terms for the classifier to make a reliable prediction. If the 
model had been forced to classify every ticket regardless of input quality, this could 
have led to incorrect issue types, misrouted tickets, and unsuitable initial priorities. 


===== PAGE 67 =====

   
 
 60  
 
This problem was addressed by adding validation logic before inference, as shown in 
Listing 5.3. The backend checks whether the text contains enough useful signal 
before passing it to the trained classifier. If not, the ticket remains uncategorised 
rather than receiving a weak prediction. 
Another challenge was keeping priority and SLA logic consistent between the 
backend and frontend. Ticket urgency in this system is not a fixed value; it depends 
on issue category, ticket age, and whether the ticket is still open or already closed, as 
shown in Listings 5.5 and 5.6. Without careful control, the same ticket could appear 
as one priority in the database logic and a different one in the user interface. This 
was solved by making the backend the primary source of truth for ticket priority and 
SLA calculation, while the frontend uses mirrored display logic so that users see the 
same values consistently throughout the system. 
A further challenge was enforcing secure role-based access, particularly after 
introducing private assist messaging. The system supports three main user roles: end 
users, technicians, and administrators. Each role requires different permissions, and 
these became more complex once messaging was extended beyond simple public 
updates. The solution was to enforce permissions primarily in the backend at both 
endpoint and query level, while also using frontend route protection through 
the RoleRoute component shown in Listing 5.2. Ticket access is checked against the 
logged-in user’s role and relationship to the ticket, and message retrieval is filtered so 
that private assist messages are returned only to authorised staff participants. To 
keep this logic consistent, shared backend helpers such 
as can_access_ticket and is_private_assist_participant, together with the 
authentication helper get_current_user, were used across protected endpoints. 
These helpers determine the logged-in user’s identity, evaluate their relationship to 
the ticket, and support the additional access rule that technicians may access tickets 
on which they are participants in a private assist exchange. Private message visibility 
is then further restricted at query level so that only authorised participants receive 
those messages. 
Managing feature growth and schema evolution was also a significant challenge. The 
project began with a relatively small scope centred on authentication and ticket 
creation, but later expanded to include automated classification, team assignment, 
notifications, attachments, dashboard analytics, private messaging, priority 
escalation, and SLA monitoring. This was addressed by keeping the implementation 
modular and extending the schema incrementally. Separate modules were 
maintained for authentication, models, schemas, and machine learning, while helper 
functions such as create_notification in Listing 5.7 were introduced for repeated 
logic to reduce duplication and limit the risk of inconsistent behaviour as the 
codebase grew. 
Overall, these challenges showed that the main implementation difficulty was not 
simply coding individual features, but integrating them into a coherent and reliable 
system. The final solution required defensive handling of weak machine learning 
inputs, careful synchronisation of business rules across backend and frontend layers, 
strict backend-enforced access control, and a modular structure capable of 
supporting continuous expansion from prototype to full application.

6. Testing and Evaluation 
6.1 Introduction 
This chapter evaluates the completed system across four dimensions: functional 
correctness, system evaluation against project aims, machine learning performance,

6.2 System Testing 
System testing was carried out to verify that the major features of the application 
worked correctly both in isolation and as part of the wider integrated workflow. 
Because the system spans a React frontend, FastAPI backend, SQLite database, and 
machine learning component, testing needed to cover multiple layers rather than 
treating each in isolation. A failure at any layer, for example, a correctly submitted 
ticket that is miscategorised due to a backend mapping error, would not be caught by 
testing a single component alone. 
Testing was performed primarily through structured functional testing of complete 
end-to-end user workflows [31]. This approach was chosen because the most 
important system behaviours emerge from component interaction rather than 
individual units. Ticket creation, for instance, exercises frontend form validation, 
backend request handling, ML inference, team mapping, priority calculation, and 
database storage in a single operation. Unit testing individual functions in isolation 
would not reveal failures in how these stages interact. 
The main areas tested were authentication and role-based access, ticket creation and 
retrieval, ML classification during submission, team routing and priority assignment, 
status and category updates, technician assignment, messaging and private assist 
messages, attachments, notification handling, dashboard views, SLA state 
calculation, and priority escalation logic.

Analysis 
The test produced correct or operationally acceptable outcomes for nine of the ten 
submitted tickets, with correct team routing achieved in all nine cases. The single 
outright failure occurred on Ticket 9, where the signal validation logic did not 
activate as designed. These results are discussed below by test category. 
Baseline Cases - Tickets 1, 4, 10 
All three baseline tickets were classified correctly and routed to the expected support 
teams with appropriate priority levels. Ticket 1 (Printer Issue), Ticket 4 (Password 
Reset), and Ticket 10 (Malware / Virus Alert) each contain highly distinctive domain-
specific vocabulary that appears with strong discriminative weight in the TF-IDF 
feature space. Terms such as "printer", "offline", "password", and "malware" are 
associated closely with a single issue category and rarely appear across others, 
allowing the classifier to identify these cases with high confidence. This is consistent 
with the per-class F1 analysis in Section 6.3.1 where these categories achieved some 
of the strongest individual scores. These results confirm that the system performs 
reliably on the clear-cut, well-described tickets that form the majority of a real 
helpdesk queue. 
Priority Verification Cases - Tickets 2, 7 
Both priority verification tickets produced correct team assignments and appropriate 
priority levels. Ticket 2 (Phishing Report) was correctly classified and assigned to the 
Security Team. Ticket 7 describing a department-wide file server outage was 
classified as Server Down / Service Outage rather than File Server Issue, which 
differs slightly from the primary expected category but remains fully acceptable. Both 
categories route to the Systems Team and both trigger a Critical priority assignment, 
meaning the operational outcome is identical regardless of which label is predicted. 
This result illustrates an important characteristic of the system design: the category-
to-team mapping provides a degree of resilience against classification at semantically 
similar boundaries, because tickets that are difficult to distinguish linguistically tend 
to belong to categories that share the same downstream routing decision. The 
business impact language in Ticket 7, specifically the reference to multiple users 
being affected and work having stopped, did not appear to directly influence the 
classification, as the classifier operates on textual patterns rather than semantic 
urgency signals. Priority assignment in this case was determined correctly by the 
issue type mapping rather than by the description content itself. 
Confusion Pair Probes - Tickets 3, 6 
These two tickets were designed specifically to target the category boundaries most 
frequently confused during synthetic dataset evaluation, and the results directly 
corroborate those findings. 
Ticket 3 described unusual account activity and logins from an unrecognised 
location. This ticket was deliberately constructed to sit at the boundary between 
Account Compromise and Suspicious Login, the most frequently confused pair in the 
confusion matrix with 49 misclassifications recorded in the synthetic evaluation. The 

===== PAGE 74 =====

   
 
 67  
 
system predicted Suspicious Login. Because the test design explicitly accepted either 
category as a valid outcome, both are operationally appropriate responses to the 
described situation and both route to the Security Team, this result is recorded as a 
pass. It is worth noting, however, that the distinction between these two categories 
does carry real operational significance in a live helpdesk environment. An Account 
Compromise implies a confirmed or likely breach requiring immediate containment, 
while a Suspicious Login is an alert requiring investigation. A system that 
consistently predicts the less severe of the two categories at this boundary could 
delay an appropriate response. This is an acknowledged limitation of the classifier at 
this specific boundary and reinforces the importance of human oversight as a 
corrective mechanism. 
Ticket 6 described a VPN connection that was timing out and disconnecting 
repeatedly. The system predicted Network Outage / Connectivity Issue rather than 
VPN Issue. The description emphasised symptoms, repeated disconnection and 
timeout behaviour, rather than the specific technology involved, which likely 
activated features associated with broader network outage terminology rather than 
VPN-specific terms. Importantly, both categories route to the Network Team, 
meaning no operational harm results from this misclassification in terms of ticket 
routing. This result is recorded as a partial outcome rather than a failure, as the 
routing decision was correct even though the category label differed from the 
primary expectation. This finding also highlights a structural characteristic of TF-
IDF classification on symptom-heavy descriptions: users who describe what 
happened rather than which system failed are more likely to produce predictions at 
category boundaries, which is a realistic pattern in real helpdesk submissions and a 
direction for future improvement. 
Ambiguous Cases - Tickets 5, 8 
Both ambiguous tickets produced operationally acceptable outcomes. Ticket 5 
describing a non-functional laptop screen with a working external monitor was 
correctly classified as Laptop/Desktop Hardware Issue and routed to Desktop 
Support. Ticket 8 describing Outlook crashing on attachment open was classified as 
Mailbox / Email Sync Issue rather than Basic Software Issue or Email Access Issue. 
While this differs from the primary expected categories, it remains defensible given 
that Outlook is an email application and the symptoms described overlap with 
mailbox behaviour. The correct team was assigned and an appropriate priority was 
set, meaning the misclassification would have minimal operational impact. Both of 
these cases illustrate that the system handles ambiguous descriptions reasonably well 
in practice, tending to predict a related and operationally sensible category rather 
than producing a completely incorrect result. 
Signal Validation Stress Test - Ticket 9 
Ticket 9 represents the only outright failure in the test and the most significant 
finding. The submission "Its not working please help" was expected to trigger the 
Uncategorized fallback, one of the key design decisions in the system introduced 
specifically to prevent low-quality descriptions from receiving a weak and potentially 
misleading prediction. Instead, the system classified this ticket as Disk Space / 
Storage Issue and routed it to Desktop Support. 
The reason for this failure is due to the signal validation logic itself. The three 
validation checks applied before inference are a minimum word count, a minimum 
unique token count, and a minimum alphanumeric character count. The submission 

===== PAGE 75 =====

   
 
 68  
 
meets all three thresholds: it contains five words, five unique tokens, and sufficient 
alphanumeric characters. All checks pass, the description is forwarded to the 
classifier, and the classifier produces a confident but entirely meaningless prediction 
because there is no domain-relevant content for it to act on. 
This exposes a fundamental limitation of the current approach: the signal validation 
checks measure the quantity of text, not the relevance of its content. A description 
can satisfy all three thresholds while containing no terminology with any 
discriminative value in the TF-IDF feature space. What is actually required is a 
measure of domain signal, for example, a check on whether any terms in the 
description overlap with the high-weight features learned during training. If no such 
overlap exists, the description should be rejected to Uncategorized regardless of its 
length. This is a straightforward improvement that would address the failure mode 
demonstrated here and represents a clear direction for future development. 
In a live deployment environment, a ticket submitted with this description would 
likely be caught during manual review given its obviously inadequate content. The 
human oversight capability built into the system, specifically the ability for 
administrators and technicians to reassign and recategorise tickets, remains an 
important safeguard at this boundary. Nevertheless, the failure of the Uncategorized 
fallback on this case is a genuine limitation of the current signal validation design. 
 
Overall Assessment 
The structured test demonstrates that the integrated system performs reliably across 
the range of ticket types most likely to be encountered in a real helpdesk 
environment. Correct or operationally acceptable outcomes were produced for nine 
of ten submissions, with correct team routing achieved in all nine of those cases. The 
single routing error on Ticket 9 was caused by a limitation in the signal validation 
logic rather than a failure of the classifier itself, and the nature of that failure is well 
understood and addressable. 
A key finding of this evaluation is the importance of distinguishing between category 
accuracy and routing accuracy as separate measures of system performance. Two of 
the tickets that produced unexpected category labels, Ticket 6 (VPN Issue predicted 
as Network Outage) and Ticket 7 (File Server Issue predicted as Server Down), still 
resulted in correct team assignment and appropriate priority levels. This is a direct 
consequence of the category-to-team mapping design, which groups semantically 
related categories under the same support team. Misclassifications at the boundaries 
between related categories, which are both the most linguistically difficult to 
distinguish and the most likely to occur, tend to produce routing decisions that 
remain operationally correct. This structural resilience is one of the more significant 
design outcomes of the project. 
The confusion pair probes confirmed the findings of the synthetic dataset evaluation. 
The Account Compromise and Suspicious Login boundary, and the VPN Issue and 
Network Outage boundary, both produced the expected confusion patterns under 
real-world conditions. This consistency between synthetic and real-world results 
provides evidence that the confusion matrix analysis in Section 6.3.1 reflects genuine 
characteristics of the classification problem rather than artefacts of the synthetic test 
data.

accuracy recorded on the full 29-category synthetic test set, where performance is 
averaged across many more category boundaries including the most difficult ones. 
Taken together, these results validate the system's core operational claim: that ML-
assisted classification can reliably route support tickets to the correct support team 
based on natural language descriptions, while human oversight remains available to 
handle edge cases and correct errors at the category level where needed. 
 
6.3 System Evaluation 
While testing confirms whether the system works correctly, evaluation considers how 
effectively the completed project addresses its original aim. 
The central aim was to design, develop, and evaluate a web-based IT ticketing system 
that uses machine learning to support the classification, routing, prioritisation, and 
management of IT support requests. This aim was achieved. The core workflows 
operate consistently across the frontend and backend, the ML classifier contributes 
directly to routing and prioritisation decisions rather than operating in isolation, and 
the role-based interface structure reduces unnecessary complexity for each user 
group by tailoring views to actual workflow responsibilities. The addition of SLA 
logic, priority escalation, notifications, and dashboard analytics moves the system 
beyond basic ticket storage toward genuine workflow support, directly addressing the 
gap identified in the literature review, where classification is treated as an isolated 
task rather than part of a broader operational process. 
The system is designed so that human oversight remains available at every stage. 
This hybrid approach, automated classification with manual correction capability, 
reflects the practical limitations identified during model evaluation and represents a 
more realistic deployment model than full automation. While a formal usability 
evaluation was not conducted, the design rationale for each role-specific view is 
grounded in the stakeholder analysis in Chapter 3 and informed by direct 
observation of real helpdesk workflows during work placement. 
There are clear limitations. SQLite and local file storage are appropriate for a 
prototype but would not scale to enterprise deployment without migration to a more 
robust database and cloud storage solution. The ML model's performance is bounded 
by the quality of the synthetic training data, and generalisation to real-world ticket 
language remains unvalidated beyond the test set. The system also lacks integration 
with enterprise services such as email platforms and identity providers. These 
limitations define the boundaries of what has been achieved rather than 
undermining it, the project is explicitly scoped as a substantial prototype, and it 
delivers meaningfully within that scope. 
 
6.3.1 Analysis of Model Performance 
The final selected model for the system, LinearSVC, achieved an accuracy of 67.31% 
and a macro F1-score of 0.6695 on an unseen test dataset comprising 7,250 ticket 
descriptions across 29 issue categories. To contextualise this result, a random 
baseline classifier would achieve approximately 3-4% accuracy on a 29-class problem 

===== PAGE 77 =====

   
 
 70  
 
[32], meaning the model represents a substantial improvement over chance-level 
performance. The macro F1-score is the primary evaluation metric used here because 
it weights each class equally regardless of support size [33], providing a balanced 
view of performance across all categories rather than favouring the most common 
ones. 
Per-class analysis reveals considerable variation in F1-score across issue types, and 
this variation is not arbitrary, it maps closely onto the semantic distinctiveness of 
each category. Categories with highly domain-specific and unambiguous vocabulary 
achieve the strongest results. Wi-Fi Connectivity Issue achieves the highest F1-score 
at approximately 0.90, followed by Printer Issue (0.87), Phishing Report (0.83), and 
Suspicious Login (0.82). These categories benefit from consistent, recognisable 
terminology, words such as "printer", "offline", "wi-fi", "wireless", and "suspicious" 
appear with high discriminative weight and rarely overlap with other categories in 
the feature space. 
In contrast, the weakest performing categories are Server Down / Service Outage (F1 
≈ 0.41), Account Compromise (F1 ≈ 0.44), and Basic Software Issue (F1 ≈ 0.46). 
Inspection of the confusion matrix explains these results. Account Compromise is 
most frequently misclassified as Suspicious Login (49 instances) and Security Policy 
Violation (20 instances). Server Down / Service Outage is most often confused with 
VM / Infrastructure Issue (46 instances) and Basic Software Issue (27 instances). 
Mailbox / Email Sync Issue (F1 ≈ 0.54) also suffers from boundary confusion, 
overlapping significantly with both Account Compromise (36 instances) and Email 
Access Issue (28 instances). 
Critically, these error patterns are not random. In each case, the misclassified 
category is operationally and linguistically related to the predicted one. A ticket 
describing an account compromise may use phrasing nearly identical to one 
reporting a suspicious login. A server outage description may mention software 
failures in the same terms as a general software issue. This semantic coherence in the 
error distribution is an important finding because it suggests the model is not failing 
unpredictably, it is making mistakes that reflect genuine ambiguity in the problem 
domain, ambiguity that would challenge a human reader encountering the same 
ticket text without additional context. 
Precision and recall analysis further illustrates this point. Access Request exhibits 
high recall (0.85) but lower precision (0.47), meaning the model correctly identifies 
most true Access Request tickets but also over-predicts this label for tickets from 
neighbouring categories such as Active Directory Issue (33 misclassifications), 
Backup / Restore Issue (26 misclassifications), and Software Installation Request (17 
misclassifications). This over-prediction likely reflects the broad, non-specific 
language users employ when submitting general access-related requests. Security 
Policy Violation similarly suffers from low precision due to overlap with categories 
such as Malware / Virus Alert and Account Compromise, where the triggering events 
and user descriptions share common vocabulary around policy, compliance, and 
account behaviour. 
Taken together, these results support two conclusions. First, the model is performing 
meaningfully on a genuinely difficult classification task, capturing discriminative 
patterns across a large label set with no feature engineering beyond TF-IDF. Second, 
and more importantly for the system design, the structure of the model's errors 
validates the decision to deploy it as a decision-support tool rather than a fully 

===== PAGE 78 =====

   
 
 71  
 
autonomous classifier. The confusion matrix shows that errors are concentrated at 
category boundaries where semantic overlap is highest, precisely the cases where 
human review adds the most value. A technician correcting a misclassification of 
"Server Down" as "VM / Infrastructure Issue" is making a small operational 
adjustment, not recovering from a fundamentally wrong prediction. This error 
profile makes the model safe and practical to integrate into a live helpdesk workflow, 
where category corrections remain possible at every stage of the ticket lifecycle. 
 
6.3.2 Dataset Considerations and Limitations 
An important factor influencing model performance is the nature of the dataset used 
for training and evaluation. Obtaining a suitable real-world dataset for this task 
presents significant practical challenges. Real helpdesk ticket data is typically 
proprietary, inconsistently labelled, and not aligned with a predefined set of issue 
categories. Using such data without extensive cleaning and relabelling would 
introduce substantial label noise, making it difficult to train a reliable supervised 
classifier or to interpret evaluation results meaningfully. 
To address this, a synthetic dataset of 43,500 labelled ticket descriptions was 
generated for training, with a separate set of 7,250 descriptions used for evaluation. 
Synthetic generation ensured controlled coverage across all 29 categories, consistent 
labelling, and a balanced class distribution that prevented the model from being 
biased toward more common issue types. It also allowed the training and test sets to 
be drawn from different generation processes, simulating generalisation to unseen 
phrasing rather than simply splitting a single homogeneous dataset. 
However, synthetic data carries inherent limitations. Generated descriptions tend to 
follow more regular structures and vocabulary than real user-submitted tickets, 
which are often abbreviated, colloquial, misspelled, or contextually ambiguous. This 
gap between training distribution and real-world input is the primary unvalidated 
risk in the current system. The model's performance of 67.31% accuracy on the 
synthetic test set provides a meaningful internal benchmark, but it does not directly 
answer the question of how the classifier would perform on tickets submitted by real 
users in a live environment. This is the most significant limitation of the machine 
learning component and represents the primary direction for future work. 
Despite this, the use of synthetic data is justified within the scope of this project. It 
enabled the construction of a well-defined, consistently labelled supervised learning 
problem that could be evaluated rigorously and integrated into the system workflow. 
A key insight from applied ML research, supported by the findings of Al-Hawari and 
Barham (2019) and Agarwal et al. (2020), is that label consistency and feature 
representation often have a greater impact on classification performance than model 
architecture[2][5]. The synthetic approach prioritised these factors, and the 
resulting classifier provides genuine operational value as a decision-support tool 
even under the acknowledged data constraints.

6.5 Limitations and Lessons Learned 
Several lessons emerge from the project that have broader relevance to applied ML 
system development.

The first is that the value of machine learning in a real application is largely 
determined by its integration context rather than its standalone performance. A 
classifier achieving 67% accuracy operating in isolation would have limited practical 
value. The same classifier embedded in a workflow where predictions drive routing 
decisions, trigger SLA timers, and can be corrected by a technician in two clicks 
becomes genuinely useful. This project demonstrates that integration design matters 
as much as model selection. 
The second lesson is that role-based systems require sustained attention to the 
distinct needs of each user group throughout design and implementation. The 
temptation to build a single general-purpose interface and add role filtering 
afterwards produces a worse result than designing each view around its primary 
user's actual workflow from the beginning. 
The third lesson concerns the trade-off between classification granularity and 
performance. Using 29 issue categories provides operationally meaningful routing 
decisions, the difference between routing to the Security Team versus the Service 
Desk matters in a real helpdesk, but it increases classification difficulty significantly. 
A 10-category model would almost certainly achieve higher accuracy, but the routing 
decisions it produces would be less actionable. This trade-off was the right one given 
the project aims, but it should be made consciously rather than discovered during 
evaluation. 
The main limitations of the project are its prototype status, the use of synthetic 
training data without real-world validation, the lack of formal usability evaluation, 
and the absence of enterprise integrations. Each of these points directly to future 
work rather than representing a failure of the current implementation.

7. Conclusions and Future Work 
7.1 Reflection on Project Aims 
The central aim of this project was to design, develop, and evaluate a web-based IT 
ticketing system that uses machine learning to support the classification, routing, 
prioritisation, and management of IT support requests. This aim was achieved. The 
completed system delivers a full ticket lifecycle from submission to closure, with 
automated issue-type prediction integrated into the ticket creation workflow, role-
based interfaces for all three user groups, and operational features including SLA 
tracking, priority escalation, messaging, attachments, notifications, and analytics 
dashboards. 
Functional testing confirmed correct behaviour across all major workflows and 
boundary conditions. The real-world classification test produced correct or 
operationally acceptable outcomes for nine of ten submitted tickets, with correct 
team routing achieved in all nine cases. The single failure, on the deliberately vague 
submission "Its not working please help", was caused by a specific and well-
understood limitation in the signal validation logic rather than a fundamental failure 
of the classification approach. 
 
7.2 Key Findings 
Routing accuracy and category accuracy are distinct measures of 
performance 
The LinearSVC classifier achieved 67.31% category-level accuracy on the synthetic 
test set, but the real-world test revealed that several tickets with unexpected category 
labels were still routed to the correct support team. A ticket predicted as Network 
Outage rather than VPN Issue still reaches the Network Team; a ticket predicted as 
Server Down rather than File Server Issue still reaches the Systems Team. This 
resilience is a direct consequence of the category-to-team mapping design, and it 
means routing accuracy is a more meaningful measure of operational performance 
than label-level accuracy for this type of system. 
 
The value of machine learning depends on integration design as much as 
model performance 
A classifier achieving 67% accuracy in isolation has limited utility. The same 
classifier embedded in a workflow where predictions drive routing decisions, trigger 
SLA timers, and can be corrected by staff delivers genuine operational value. The 
design decisions that contributed most were integration decisions: the Uncategorized 
fallback to an Admin Review Queue, staff correction capability, and the separation of 
category prediction from priority assignment so that corrections propagate correctly 
through downstream logic. 
 
Signal validation measures quantity rather than domain relevance 
The evaluation identified a specific limitation in the signal validation logic. The three 
checks applied before inference are designed to prevent low-quality descriptions 

===== PAGE 82 =====

   
 
 75  
 
from receiving weak predictions, but the Ticket 9 test case demonstrated that a 
description can satisfy all three checks while containing no terminology with 
discriminative value. The submission "Its not working please help" passed all checks 
and received a confident but meaningless prediction. The root cause is that the 
checks measure the quantity of text rather than its relevance to the domain. The most 
impactful near-term technical improvement would be augmenting these checks with 
a domain signal measure, such as a minimum overlap threshold between the 
description's terms and the high-weight features in the trained TF-IDF vocabulary. 
 
7.3 Limitations 
The limitations of this project are acknowledged here and ranked by their 
significance to the overall system. 
The most significant limitation is the reliance on synthetic training data. The 
classifier was trained and evaluated on synthetically generated ticket descriptions, 
which tend to follow more regular structures and vocabulary than real user-
submitted tickets. The model's performance on the synthetic test set provides a 
meaningful internal benchmark but cannot confirm how it would perform on tickets 
submitted by real users in a live environment. This is the primary unvalidated risk in 
the current system. 
The second most significant limitation is the absence of a formal usability evaluation. 
The system was designed with distinct role-based interfaces grounded in stakeholder 
analysis and placement observation, but no users were observed using the system 
and no structured feedback was gathered. The claim that the interface is usable in 
practice therefore rests on design reasoning rather than empirical evidence. 
Further limitations include the use of SQLite and local file storage, which are 
appropriate for a prototype but would not support concurrent multi-user access at 
organisational scale, and the absence of enterprise integrations such as identity 
providers or email platforms, which limits practical deployment readiness. 
 
7.4 Future Work 
Future work is presented in priority order based on the evaluation findings rather 
than feature desirability alone. 
The highest priority is validating and improving the machine learning component 
against real-world data. This involves collecting labelled ticket descriptions from a 
real helpdesk environment and evaluating the current classifier against this data to 
understand the gap between synthetic and real-world performance. Depending on 
the size of the available dataset, this could also support retraining the model on real 
data or implementing active learning to improve the classifier continuously as new 
tickets are resolved. 
The second priority is improving the signal validation logic. Replacing or augmenting 
the current checks with a domain signal measure would directly address the failure 
mode demonstrated by Ticket 9 and reduce the rate of low-confidence predictions 
incorrectly passed to the classifier.

 76  
 
The third priority is conducting a formal usability evaluation. Recruiting a small 
number of participants representing each user role, defining structured task 
scenarios, and collecting both quantitative and qualitative feedback would provide 
empirical support for the interface design decisions and identify areas for 
improvement. 
Beyond these priorities, future development could address production readiness 
through migration to PostgreSQL, cloud-based attachment storage, and 
containerised deployment. Integration with enterprise systems, security hardening, 
advanced reporting, and mobile interface optimisation represent further directions 
that would improve the system's suitability for real organisational use. 
 
7.5 Final Conclusion 
The aim of this project was to design, develop, and evaluate an intelligent IT ticketing 
system using machine learning to support ticket classification, routing, prioritisation, 
and management. This was achieved through a full-stack web application integrating 
a React frontend, FastAPI backend, SQLite database, and a TF-IDF and LinearSVC 
pipeline embedded directly in the ticket creation workflow. 
The evaluation confirmed reliable behaviour across all core workflows, with correct 
or operationally acceptable outcomes for nine of ten real-world test submissions and 
correct team routing in every predicted case. A key finding was that routing accuracy 
consistently exceeded category-level accuracy due to the team mapping design, 
demonstrating that integration design provides meaningful resilience against 
classification errors in practice. 
The use of synthetic training data and the absence of formal usability testing are 
acknowledged limitations, both identified as priorities for future development. 
Within these boundaries, the project demonstrates that machine learning can be 
embedded practically and effectively into a real IT support workflow, contributing 
directly to routing and prioritisation decisions while retaining human oversight at 
every stage.

category from the ticket description. If the text is too weak, the system 
returns Uncategorized instead of forcing a prediction. The predicted category is then 
used to support team routing and priority estimation.

The machine learning component of this system was trained and evaluated using a

The dataset consists of 43,500 labelled ticket descriptions across 29 issue categories,

The synthetic descriptions were generated to reflect the variety of phrasing styles 
encountered in real helpdesk environments. Rather than using uniform template-

• Escalation notes: descriptions prefixed with escalation context, simulating

• Follow-up entries: descriptions representing updates added to an existing 
ticket thread 
• Short requests: brief, minimal descriptions simulating users who provide 
little detail 
• Desk triage notes: descriptions written from the perspective of a helpdesk

C.5 Train and Test Split Strategy 
A decision was made not to use a standard random train/test split from a single 
dataset. Instead, the training set of 43,500 descriptions and the evaluation set of 
7,250 descriptions were generated through separate processes. This approach was 
chosen to simulate generalisation to genuinely unseen phrasing rather than simply 
holding out a portion of identically generated data. 
If both sets had been drawn from the same generation process, the model would 
achieve inflated evaluation scores by recognising structural patterns in the generated 
text rather than learning the underlying characteristics of each issue type. Using 
separate generation processes means the evaluation set contains different sentence 
structures, vocabulary choices, and stylistic patterns from the training set, providing 
a more meaningful test of how well the model generalises. 
 
C.6 Limitations of the Synthetic Dataset 
The use of synthetic data introduces acknowledged limitations that are important to 
state clearly: 
Vocabulary regularity: Synthetic descriptions tend to follow more consistent 
grammatical structures and vocabulary than real user-submitted tickets, which are 
often abbreviated, colloquial, misspelled, or highly context-dependent. 
Missing noise: Real helpdesk tickets frequently contain typos, incomplete 
sentences, irrelevant personal context, and ambiguous phrasing that synthetic 
generation does not fully replicate. 
Unknown generalisation gap: The model's performance of 67.3% accuracy on 
the synthetic test set provides a meaningful internal benchmark but cannot confirm 
how the classifier would perform on tickets submitted by real users in a live 
environment. This gap between synthetic evaluation performance and real-world 
performance is the primary unvalidated risk in the machine learning component. 
No real-world validation: Obtaining a suitable real-world labelled dataset was 
not feasible within the scope of this project. Real helpdesk ticket data is proprietary, 
subject to data protection constraints, and not publicly available in a labelled form 
aligned with the 29 categories used in this system. This is a field-wide constraint 
rather than a limitation specific to this project, and real-world data validation is 
identified as the highest priority direction for future work in Chapter 7.

Appendix F: Glossary of Terms 
The following definitions are provided for readers who may be less familiar with the 
machine learning concepts used in this project. 
Unstructured Ticket Data Support request information that exists as free-form 
natural language text rather than in a predefined, machine-readable format. When a 
user types a description such as "my laptop keeps freezing when I open Chrome", 
that text has no fixed fields, consistent vocabulary, or guaranteed format. This 
contrasts with structured data, such as a dropdown where the user selects a category 
from a predefined list. Unstructured data is harder for a system to process 
automatically, which is the core problem this project addresses. 
Machine Learning Classifier A type of algorithm that learns to assign labels or 
categories to new inputs based on patterns identified in previously labelled examples. 
In this project, the classifier is trained on thousands of labelled ticket descriptions 
and then used to predict the issue type of newly submitted tickets it has never seen 
before. 
Supervised Learning A type of machine learning in which the model is trained on 
a labelled dataset, that is, a collection of examples where the correct answer is 
already known. The model learns the relationship between inputs and outputs during 
training and can then make predictions on new, unseen inputs. This project uses 
supervised learning because each training ticket description is paired with a known 
issue type label. 
TF-IDF (Term Frequency – Inverse Document Frequency) A technique for 
converting text into numerical values that a machine learning model can process. It 
assigns a weight to each word based on two factors: how often the word appears in a 
specific document (term frequency), and how rare it is across all documents (inverse 
document frequency). Words that are distinctive to a particular ticket type, such as 
"VPN" or "printer", receive high weights, while common words like "help" or "issue" 
receive low weights. In this project the vectoriser is configured to extract both 
individual words and two-word phrases (bigrams), as many IT issue types are better 
described by short phrases than by single words alone. 
Token / Tokenisation A token is an individual unit of text, typically a word or 
short phrase, that a model processes. Tokenisation is the process of splitting raw text 
into these units before further processing. In this project, the TF-IDF vectoriser 
tokenises ticket descriptions into individual words and two-word phrases before 
converting them into numerical feature vectors. 
Vocabulary In the context of text classification, the vocabulary is the complete set 
of unique words and phrases that the TF-IDF vectoriser has learned from the 
training data. Each entry in the vocabulary corresponds to one dimension in the 
feature vector. The size of the vocabulary determines how many features the model 
works with, which is why ticket text produces high-dimensional sparse vectors. 
Feature Vector A numerical representation of an input, in this case, a ticket 
description, used by a machine learning model. After TF-IDF processing, each ticket 
is converted into a list of numbers where each number represents the weight of a 
particular word or phrase. The model uses these vectors rather than raw text to make 
predictions. 

===== PAGE 105 =====

   
 
 98  
 
Sparse Vector / High-Dimensional Data A feature vector is described as sparse 
when most of its values are zero. In text classification, the vocabulary across all 
tickets may contain thousands of words, but any single ticket will only use a small 
number of them. The resulting vector therefore has thousands of dimensions but 
only a handful of non-zero values. LinearSVC is particularly well suited to this type of 
data. 
ML Pipeline A sequence of processing steps chained together so that the output of 
one step becomes the input of the next. In this project, the pipeline consists of two 
stages: the TF-IDF vectoriser, which converts raw ticket text into numerical feature 
vectors, followed by the LinearSVC classifier, which predicts the issue type from 
those vectors. Packaging these steps as a single pipeline ensures that the same 
preprocessing is applied consistently during both training and prediction. 
Training Data / Test Data In machine learning, the available labelled data is 
typically split into two subsets. Training data is used to teach the model the patterns 
it needs to make predictions. Test data is a separate, unseen subset used to evaluate 
how well the trained model performs on examples it was not trained on. This split is 
important because a model that simply memorises the training data may perform 
poorly on new inputs, a problem known as overfitting. 
Baseline / Benchmark A baseline is a simple reference result against which more 
sophisticated models are compared. In this project, Logistic Regression served as a 
strong baseline model. A model that cannot outperform the baseline offers little 
justification for its added complexity. Benchmark refers more broadly to an 
established standard of comparison, such as the accuracy scores reported in prior 
academic work. 
Logistic Regression A supervised classification algorithm that estimates the 
probability of an input belonging to a particular category. Despite its name, it is a 
classification method rather than a regression one. It learns a set of weights for each 
feature and uses these to calculate a probability score. In this project it served as a 
strong baseline model against which other classifiers were compared. 
Naive Bayes A probabilistic classification algorithm based on Bayes' theorem, 
which calculates the likelihood of a category given the observed features. It is 
described as "naive" because it assumes that all features are statistically independent 
of each other, an assumption that rarely holds in natural language but still produces 
reasonable results in many text classification tasks. It was evaluated as one of the five 
candidate models in this project. 
SGDClassifier A linear classifier trained using Stochastic Gradient Descent, an 
optimisation technique that updates the model incrementally on small batches of 
data rather than the entire dataset at once. It can approximate the behaviour of 
Logistic Regression or a linear SVM depending on its configuration. In this project it 
achieved the highest raw accuracy score but was not selected for deployment due to 
concerns about consistency compared to LinearSVC. 
Random Forest An ensemble learning method that builds a large number of 
decision trees during training and combines their outputs to make a final prediction. 
Each tree is trained on a random subset of the data and features, reducing the risk of 
overfitting. While powerful in many settings, Random Forest is generally less suited 
to high-dimensional sparse text data, which is reflected in its lower performance in 
this project's model comparison. Ensemble methods more broadly combine 

===== PAGE 106 =====

   
 
 99  
 
predictions from multiple models for greater robustness, but were found in this 
project to offer only marginal gains over individual linear classifiers at significantly 
higher computational cost. 
LinearSVC A type of Support Vector Machine classifier that finds the optimal 
boundary between categories in high-dimensional space by identifying the 
hyperplane that best separates one class from another with the largest possible 
margin. LinearSVC performs particularly well on sparse, high-dimensional text data 
and is computationally efficient, making it suitable for real-time prediction within a 
web application. 
Hyperparameter A configuration setting of a machine learning model that is set 
before training begins, as opposed to parameters that the model learns from the data 
itself. Examples include the regularisation strength of a classifier or the maximum 
number of features in a vectoriser. In this project, extensive hyperparameter 
optimisation was deliberately avoided in favour of simpler, well-validated defaults 
supported by the literature. 
Inference The process of using a trained machine learning model to make 
predictions on new, unseen inputs, as distinct from training. In this project, 
inference happens in real time whenever a user submits a ticket, the trained pipeline 
receives the ticket description and returns a predicted issue type within the same web 
request. 
Overfitting / Generalisation Overfitting occurs when a model learns the training 
data too closely and as a result performs well on training examples but poorly on 
new, unseen data. Generalisation refers to the model's ability to perform reliably on 
new inputs by capturing underlying patterns rather than memorising specific 
examples. In this project, the model was evaluated on a separate unseen test set to 
measure its generalisation ability. 
Accuracy The proportion of predictions the model got correct out of all predictions 
made. While intuitive, accuracy alone can be misleading when categories are 
imbalanced. In this project, macro F1-score was prioritised over accuracy to ensure 
all 29 issue types were treated equally in the evaluation. 
Macro F1-Score A performance metric used to evaluate classification models 
across multiple categories. It calculates the F1-score (a balance between precision 
and recall) for each category individually, then averages those scores equally 
regardless of how many examples exist per category. This makes it particularly 
appropriate for this project's 29 issue types, as it treats all categories as equally 
important rather than allowing large categories to dominate the overall score. 
Precision and Recall Two complementary measures of classifier performance. 
Precision measures how often the model is correct when it predicts a particular 
category. Recall measures how often the model correctly identifies all instances of a 
category. A classifier can be high in one and low in the other; F1-score summarises 
both into a single value. 
Confusion Matrix A table used to visualise the performance of a classifier across 
all categories. Each row represents the actual category of a ticket and each column 
represents what the model predicted. The diagonal shows correct predictions; off-
diagonal values show misclassifications. In this project, the confusion matrix 
revealed which category pairs were most frequently confused, such as Account 

===== PAGE 107 =====

   
 
 100  
 
Compromise and Suspicious Login, informing the design of the real-world 
classification test. 
Natural Language Processing (NLP) A field of computer science and artificial 
intelligence concerned with enabling computers to understand and interpret human 
language. In this project, NLP techniques, specifically TF-IDF vectorisation, are used 
to convert unstructured ticket descriptions into a format that a machine learning 
model can process and classify. 
Transformer / BERT Transformers are a class of deep learning model architecture 
that have become dominant in natural language processing. BERT (Bidirectional 
Encoder Representations from Transformers) is one widely known example. These 
models can capture complex contextual relationships in text but require significantly 
more computational resources and implementation complexity than classical 
approaches. In this project, transformer-based models were considered but not used, 
as classical TF-IDF and LinearSVC pipelines achieve comparable performance on 
short helpdesk ticket text at a fraction of the cost. 
Scikit-learn A widely used open-source Python library for machine learning. It 
provides efficient implementations of classification, regression, and clustering 
algorithms, as well as tools for preprocessing, model evaluation, and pipeline 
construction. In this project, scikit-learn was used to train and evaluate all five 
candidate classifiers, construct the TF-IDF and LinearSVC pipeline, and serialise the 
trained model for deployment. 
Joblib / Model Serialisation Joblib is a Python library used to save a trained 
machine learning model to disk and reload it later. In this project, once the TF-IDF 
and LinearSVC pipeline was trained, it was saved to a .pkl file using joblib. The 
FastAPI backend loads this file once at startup and reuses it for every subsequent 
classification request, avoiding the need to retrain the model each time the 
application runs. 
Uncategorized Fallback A design decision whereby tickets containing insufficient 
or ambiguous text are not forced through the classifier. Instead of producing a low-
confidence and potentially incorrect prediction, the system assigns the ticket to an 
Admin Review Queue for manual triage. This prevents poorly written descriptions 
from being automatically misrouted and ensures that human oversight remains 
available when the classifier cannot make a reliable prediction.
